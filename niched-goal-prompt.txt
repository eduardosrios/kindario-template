## Tooling Rule

Use Product Design plugin only.

Do not use non-Product Design skills for design, UI/UX, frontend, copywriting, SEO, content strategy, accessibility, review, or writing.

Product Design is allowed only as a helper for interpreting selected screenshots and translating them into a faithful static implementation.

## Screenshot Authority

`screenshots-references/` is the primary design authority.

The future page must be assembled from screenshot-derived parts, not from a default charity layout.

Select 40 usable screenshot references when available.

Copy selected references into `used/`.

Create `used/manifest.md` with a detailed map from references to implementation decisions.

## Required Visual Extraction

Before any HTML/CSS/JS is written, extract and cite screenshot filenames for:

- header/navigation;
- hero composition;
- seven-section order;
- campaign/support section structure;
- donor/support CTA treatment;
- volunteer path treatment;
- story/field-note treatment;
- trust/transparency treatment;
- footer structure;
- typography;
- color;
- image crop;
- card style;
- form/control style;
- responsive stacking.

No major decision may come from generic memory or previous local template structure.

## Anti-Generic Project Requirements

Do not reuse the previous Kindario page structure:

- avoid same centered logo-left/nav-center/CTA-right top bar unless screenshots prove that exact treatment is best;
- avoid same dark-green split hero unless screenshots selected for this run require it;
- avoid same card grid rhythm;
- avoid same support form placement;
- avoid same section order;
- avoid same visual pacing.

The future result must visibly look like a new screenshot-led composition.

## Content Constraints

- English only.
- Do not use the word `template` in visible site copy.
- Do not invent donation totals, awards, partners, certifications, countries served, outcomes, or active programs.
- Do not create fake transaction or application behavior.
- Write dignity-first humanitarian copy.
- Keep metadata truthful and not stuffed with keywords.

## Future Static Build Scope

Build only inside `template/`.

Use HTML, CSS, JavaScript, local images, local fonts, and local vendor assets only when they support the selected screenshot direction.

Do not use React, Next.js, Vue, Angular, Vite, backend, database, admin panels, CMS code, or fake backend processing.

## Required Page Structure

- Header/navigation.
- Hero.
- Exactly seven main sections after hero.
- Footer.

Each major section must trace to at least two selected screenshots:

- primary structural reference;
- secondary detail reference.

## Definition Of Done

The work is not done until evidence proves:

- 40 references selected when available;
- selected references copied into `used/`;
- `used/manifest.md` explains every selected reference;
- visual extraction cites screenshot files;
- implementation is built inside `template/`;
- one H1 exists;
- exactly seven main sections after hero exist;
- browser screenshots were captured at 375, 430, 768, 1024, 1366, and 1440 widths;
- no broken assets or console errors;
- no horizontal overflow or clipped text;
- interactions work honestly;
- final visual comparison against `used/` shows clear screenshot-derived structure;
- result does not resemble the previous generic template.
