# Generic Goal Prompt For Any Static Front Page

Build a static front page only when explicitly asked.

Before implementation, read:

- `project_brand.md`
- `prompt5.txt`
- `AGENTS.md`
- `design.md`
- `prd.md`
- `PLAN.md`
- `roadmap.md`

## Tooling

Use Product Design plugin only.

Do not use any non-Product Design skill, including design, UI/UX, frontend, SEO, copywriting, content strategy, review, accessibility, or writing skills.

If any non-Product Design skill is available, ignore it.

## Visual Source Of Truth

`screenshots-references/` is the dominant visual source of truth.

Do not use the screenshots as loose inspiration. Use them as section-level source material.

Select 40 usable screenshot references when at least 40 exist.

If fewer than 40 usable references exist, select every usable reference and report the limitation before implementation.

## Required Reference Workflow

1. Visually inspect `screenshots-references/`.
2. Inspect `dont-create-like-this/` if present.
3. Select the strongest 40 usable references.
4. Empty stale files from `used/` unless intentionally reused.
5. Copy selected references into `used/`.
6. Create or update `used/manifest.md`.
7. Write a visual extraction before implementation.
8. Build from the extraction.
9. Compare final rendered screenshots against `used/`.

## Manifest Requirements

`used/manifest.md` must explain:

- selected file list;
- exact section or component influenced by each file;
- primary structural references;
- secondary detail references;
- rejected generic references when relevant;
- how the final design avoids a generic skeleton.

## Visual Extraction Requirements

Before writing HTML/CSS/JS, extract:

- header/navigation treatment;
- hero composition;
- exact seven-section rhythm;
- typography scale and font pairing;
- CTA treatment;
- button behavior;
- card composition;
- form/input treatment;
- image cropping and scale;
- overlay rules;
- icon style;
- color palette;
- contrast strategy;
- spacing system;
- grid behavior;
- mobile stacking pattern;
- footer structure;
- animation and interaction style.

Every extracted decision must cite screenshot filenames.

If a decision cannot be tied to screenshots, remove it or justify it as a technical requirement.

## Anti-Generic Requirement

Do not build from:

- generic hero plus card grid;
- standard landing-page skeleton;
- default Product Design output;
- Bootstrap starter page;
- previous local template structure;
- skill-generated fallback.

Every major section must trace to at least two selected screenshots:

- one primary structural reference;
- one secondary detail reference.

## Implementation Scope

Build only inside `template/`.

Use:

- HTML;
- CSS;
- JavaScript;
- local images;
- local fonts;
- local vendor assets only when needed.

Do not use:

- React;
- Next.js;
- Vue;
- Angular;
- Vite;
- backend;
- database;
- admin panel;
- CMS code;
- fake payment, booking, login, account, or backend submission behavior.

## Page Structure

The front page must include:

- header/navigation;
- hero;
- exactly seven main sections after hero;
- footer.

Do not count nested cards, overlays, columns, or subsections as main sections.

Visible site copy must not mention the word `template`.

## Required QA

Verify with browser screenshots or Playwright:

- 375px;
- 430px;
- 768px;
- 1024px;
- 1366px;
- 1440px.

Definition of Done:

- one H1;
- exactly seven main sections after hero;
- no horizontal overflow;
- no broken local assets;
- no missing images;
- no overlapping text;
- no clipped CTAs;
- working header/navigation;
- working mobile navigation;
- working interactions;
- visible focus;
- reduced-motion support for heavy animation;
- no console errors;
- visual comparison against `used/` proves screenshot influence.

Do not call implementation complete if final screenshots still look generic.
